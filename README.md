# SalesDB: Multi-Channel Sales Reconciliation & Supply Chain Analytics

![Database Relationship Diagram](relationship_diagram.png)

## 📌 Project Overview
This project delivers an **operational SQL database solution** engineered for a multi-location **smoothie chain** to bridge the gap between B2C retail POS sales and B2B distributor logistics. 

By unifying disparate data sources, this database serves as a single source of truth to drive multi-channel sales reconciliation, master data alignment, and distributor ordering cycle forecasting.

---

## 🗂️ Core Database Schema & Data Dictionary

The database consists of the following key tables, engineered with clean relational constraints and data governance principles:

### 1. `Store` (Dimension Table)
The master registry of all store locations. It maps internal retail storefront names to official distributor identification codes.
*   **`OurName`** (Text / PK): The location name as generated by the retail POS system.
*   **`StoreCode`** (Text): The official B2B store identification code, borrowed directly from **Gordon Food Service (GFS)** to enable seamless cross-channel joins.

### 2. `Oursales_items` (Fact Table)
Raw transactional data from the retail POS system, capturing main product sales volume.
*   **`Location`** (Text): Retail storefront name (foreign key to `Store.OurName`).
*   **`ItemCode`** (Text): The product code, mapped and standardized using **GFS item numbers** for supply chain alignment.
*   **`Month`** (Text): Transactional period (formatted as `YYYY-MM`).
*   **`Qty`** (Double): Quantity sold at retail.
*   **`StoreCode`** (Text): Operational placeholder updated via data preparation scripts.

### 3. `Oursales_modifiers` (Fact Table)
Transactional data from the retail POS system capturing custom add-ons, toppings, and item modifications selected by customers.
*   **`Location`** (Text): Retail storefront name (foreign key to `Store.OurName`).
*   **`ItemCode`** (Text): The standardized product code (aligned with **GFS item numbers**).
*   **`Month`** (Text): Transactional period (formatted as `YYYY-MM`).
*   **`Qty2`** (Double): Modification quantity sold.
*   **`StoreCode`** (Text): Operational placeholder updated via data preparation scripts.

### 4. `GFS_Data` (Fact Table)
B2B wholesale distribution and fulfillment data, tracking physical product shipments.
*   **`StoreCode`** (Text): The GFS store code (foreign key to `Store.StoreCode`).
*   **`ItemCode`** (Text): The official GFS item identifier.
*   **`Month`** (Text): Fulfillment period (formatted as `YYYY-MM`).
*   **`ShipQty`** (Double): Actual wholesale volume shipped to the location.

## 🚀 Business Applications & Impact

This analytics engine transitions operational workflows from manual spreadsheet checking to automated SQL-driven insights:

1.  **Cross-Channel Quantity Reconciliation**: Merges retail-level sales data with wholesale distributor shipment volumes to monitor inventory flow accuracy across channels.
2.  **Automated Master Data Governance**: Implements automated exception monitoring to flag newly opened stores missing from the master registry, ensuring 100% data compliance before analysis.
3.  **B2B Order Cycle Forecasting**: Simulates window functions using advanced relational queries to calculate historical ordering frequency (order gaps) per item per store, directly optimizing supply chain stock management.
